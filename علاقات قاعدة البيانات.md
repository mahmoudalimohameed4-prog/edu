# EduSwap Database Relationship Model

This document outlines the entities and relationships within the `eduswap_fixed` database, detailing the types of connections and their business logic.

## Entities and Descriptions

### Core Entities
- **Users**: Individuals who can buy, sell, or both. Stores profile, rating, and verification status.
- **Items**: Products listed for sale or exchange. Linked to a category and a seller.
- **Categories**: Taxonomies for grouping items (e.g., Textbooks, Electronics).

### Interaction Entities
- **Orders**: Transaction records between a buyer and a seller for one or more items.
- **Chats**: Text-based communication between users.
- **Ads**: Promotional listings for items.
- **Reviews**: Feedback and ratings left by users for other users after a transaction.
- **Favorites**: List of items marked as favorites by users.

### Metadata Entity (MOS - Metadata/Operational Support)
- **Types**: Defines user roles (Seller, Buyer, Both).
- **Item Images**: Stores multiple images for each item.
- **OTP Codes**: Managed operational data for email verification.

## Relationship Map

### 1:N (One-to-Many) Relationships
| Parent Entity | Child Entity | Relationship Type | Description |
| :--- | :--- | :--- | :--- |
| **Users** | **Items** | 1:N | One user can list many items for sale. |
| **Users** | **Orders (Buyer)** | 1:N | One user can place many orders as a buyer. |
| **Users** | **Orders (Seller)** | 1:N | One user can receive many orders as a seller. |
| **Users** | **Reviews (Given)** | 1:N | One user can write many reviews. |
| **Users** | **Reviews (Received)** | 1:N | One user can receive many reviews from others. |
| **Users** | **Chats (Sent/Received)** | 1:N | One user can participate in many chat messages. |
| **Categories** | **Items** | 1:N | One category can contain many different items. |
| **Items** | **Item Images** | 1:N | One item can have multiple images associated with it. |
| **Orders** | **Order Items** | 1:N | One order can consist of multiple individual items. |
| **Types** | **Users** | 1:N | Each user is assigned one specific type/role. |

### M:N (Many-to-Many) Relationships
| Entity A | Entity B | Junction/Middle Entity | Description |
| :--- | :--- | :--- | :--- |
| **Users** | **Items** | **Favorites** | Many users can favorite many items. |
| **Orders** | **Items** | **Order Items** | Many orders can contain many items (and vice-versa in history). |

## Relationship Details and Logic

### 1. User Engagement (1:N)
- **Seller -> Items**: The `seller_id` in `items` links to `Us_id`. If a user is deleted, their items are removed (`ON DELETE CASCADE`).
- **User -> Types**: Standard lookup table `types` defines if a user is a buyer, seller, or both.

### 2. Transaction Flow (1:N & 1:1)
- **Buyer/Seller -> Orders**: Orders track both parties. Statuses range from `pending` to `delivered`.
- **Orders -> Order Items**: Breaks down the `Order_total` into individual `Item_price` and `Item_subtotal`. A trigger automatically calculates subtotals on insert.

### 3. Feedback and Reputation
- **Reviews**: Linked to `reviewer_id` and `reviewed_user_id`. Also optionally linked to an `Order_id` and `It_id` to provide context for the review. A trigger automatically updates the `Us_rating` in the `users` table after a review is inserted.

### 4. Communication
- **Chats**: Tracks `sender_id` and `receiver_id`. The application logic typically groups these into conversations based on these two IDs.

### 5. Advertisements
- **Ads**: Links a specific `It_id` (Item) and `Us_id` (User/Owner) to set promotional durations and track performance (`Ads_impressions`, `Ads_clicks`).

## MOS (Metadata/Operational Support) Summary
The **MOS** layer in this database consists of:
- **`types`**: Provides standardized roles.
- **`otp_codes`**: Ensures secure email verification flow.
- **`item_images`**: Decouples image storage from the main item entity for better performance and scalability (allows 1:N images).
- **`view_active_items` / `view_user_stats`**: These views act as operational support for quick reporting and dashboarding without complex JOINs in the application layer.
